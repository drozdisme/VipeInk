## Fresh findings

### 1. TON market reality favors simple wallet + stablecoin payment flows over zk-heavy MVP
Source type: web search, last 7 days + current ecosystem pages

Relevant signals:
- Tonkeeper and TON Wallet already train user expectations around self-custody, transfers, QR/payments, and simple token flows.
- MyTonWallet positions open-source/self-custodial wallet UX as a trust factor.

Implication:
- MVP should NOT try to compete on wallet breadth.
- MVP should focus on a narrow payment/use-case layer on top of existing TON wallet behavior.

### 2. Competitor direction is toward managed stablecoin settlement and fiat abstraction
Recent signals:
- Circle launched CPN Managed Payments (Apr 8) for institutions, abstracting blockchain complexity behind fiat workflows.
- TON + SCRYPT partnership highlights institutional access to USDT on TON.

Implication:
- product value should likely be in orchestration, user simplicity, and settlement workflow, not in building a novel chain stack first.

### 3. Current architecture note is far beyond 2-4 week MVP scope
Observed in [[Architecture]]:
- ZK-rollup
- sequencer committee
- proof marketplace
- state root machinery
- bridge and DA layers
- compliance layer
- ATM / fiat integrations

Implication:
- this is long-term architecture or vision, not MVP.
- for 1 developer, $0 budget, <2-4 weeks, most of this must be deferred.

## Decision summary
Recommended MVP direction:
- Keep long-term vision: programmable fiat/stablecoin settlement on TON
- Narrow MVP to a thin execution layer:
  - user payment intent
  - wallet handoff/signing
  - transaction preparation
  - basic transfer / swap / settlement tracking
  - simple proof-of-completion / status logging

Rejected for MVP now:
- custom ZK rollup
- custom prover market
- L1/L2 bridge implementation
- ATM integrations
- bank API integrations
- institutional compliance stack
- own wallet from scratch

Reason:
- too much surface area
- high implementation risk
- zero-budget constraint
- weak fit for early validation

## Recommended MVP question
The most useful validation question is likely:
- Can we make TON-based stablecoin payment initiation and confirmation dramatically simpler for a narrow user segment?
